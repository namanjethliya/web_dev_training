01

Sync vs Async — Why it Matters

JavaScript is single-threaded — it can only do one thing at a time. Understanding the difference between synchronous and asynchronous code is the foundation of everything in this guide.

Synchronous — Blocking

🚦

In synchronous code, each line runs one after the other. The next line cannot start until the current one finishes. If one line takes 5 seconds, everything stops for 5 seconds.

Synchronous — each line blocks the next
console.log("Step 1");          // runs immediately
console.log("Step 2");          // runs only after Step 1 finishes
console.log("Step 3");          // runs only after Step 2 finishes

// Output (always in this order):
// Step 1
// Step 2
// Step 3

The Problem — What if One Step is Slow?

⚠️

Imagine fetching data from a server takes 3 seconds. In synchronous code, the browser would freeze completely — no scrolling, no clicking, no animation — for those 3 seconds. That's unacceptable in a real app.

The Restaurant Analogy

Synchronous Waiter (bad)

Takes your order. Walks to the kitchen. Stands there waiting while the chef cooks. Walks back. Takes the next table's order. Every table waits for the previous table's food.

Asynchronous Waiter (good)

Takes your order. Gives it to the kitchen. Immediately goes to the next table. Kitchen calls "order ready!" and the waiter picks it up. Many tables served at once — nobody waits for someone else's food.

Asynchronous — Non-Blocking

In asynchronous code, slow tasks (timers, network calls, file reads) are handed off to the browser's background system. Your code continues running immediately and the result is delivered later via a callback, Promise, or async/await.

Asynchronous — slow task does NOT block
console.log("Step 1 — start");

// setTimeout is async — it fires after 2000ms
// but code does NOT pause here
setTimeout(() => {
  console.log("Step 2 — after 2 seconds");
}, 2000);

console.log("Step 3 — runs IMMEDIATELY, before Step 2");

// Output:
// Step 1 — start
// Step 3 — runs IMMEDIATELY, before Step 2
// Step 2 — after 2 seconds  (2 seconds later)

What JavaScript Hands Off to the Browser

Timers

setTimeout, setInterval — the browser's timer runs in the background.

Network Requests

fetch(), XMLHttpRequest — the browser's network stack fetches data while JS continues.

User Events

Click, scroll, keypress — the browser listens in the background and fires your callbacks.

File / DB Operations

Reading files, IndexedDB queries — handed off to the OS or browser storage engine.

Key insight: Async doesn't mean "runs in parallel threads." JavaScript is still single-threaded. It means "start the task, continue with other work, handle the result when it's ready." The browser's Web APIs do the actual waiting.

setTimeout & setInterval

The two most fundamental async tools in JavaScript — schedule code to run once after a delay, or repeatedly on an interval.

setTimeout — Run Once After a Delay

Syntax: setTimeout(callback, delayInMs)
// Run a function once after 2000ms (2 seconds)
setTimeout(() => {
  console.log("This runs after 2 seconds");
}, 2000);

// With a named function
function greet() {
  console.log("Hello from the future!");
}
setTimeout(greet, 1000);   // 1 second delay

// 0ms delay — still async! Runs after current synchronous code
setTimeout(() => {
  console.log("I run 2nd, even though delay is 0");
}, 0);
console.log("I run 1st");

clearTimeout — Cancel a Pending Timer

app.js
// setTimeout returns a timer ID you can use to cancel it
const timerId = setTimeout(() => {
  console.log("This will NEVER run");
}, 5000);

// Cancel it before it fires
clearTimeout(timerId);
console.log("Timer cancelled!");

setInterval — Repeat on an Interval

Syntax: setInterval(callback, intervalInMs)
// Runs every 1000ms (1 second) forever — until cleared
let count = 0;
const intervalId = setInterval(() => {
  count++;
  console.log("Tick:", count);

  if (count === 5) {
    clearInterval(intervalId);   // stop after 5 ticks
    console.log("Interval stopped.");
  }
}, 1000);
⏱️

Sandbox limitation with async timers: The sandbox below uses eval() which runs synchronously. Because setTimeout is async, its callback fires after eval() completes — at a point when the sandbox's console override has already been restored. This means timer callbacks may print to the real browser console (F12) instead of the sandbox output panel.

Callbacks & Callback Hell

A callback is a function passed as an argument to another function, to be called later. It was JavaScript's original async pattern — powerful but painful when nested.

What is a Callback?

A callback is just a function passed to another function
// greet() receives a callback and calls it
function greet(name, callback) {
  console.log("Hello, " + name + "!");
  callback();   // call the function that was passed in
}

function sayGoodbye() {
  console.log("Goodbye!");
}

greet("Naman", sayGoodbye);
// Output:
// Hello, Naman!
// Goodbye!

// More commonly written with an arrow function inline:
greet("Naman", () => {
  console.log("Goodbye!");
});

Async Callbacks — The Original Pattern

Simulating an async task with setTimeout
// Simulate fetching a user from a server (takes 1 second)
function fetchUser(id, callback) {
  console.log("Fetching user", id, "...");
  setTimeout(() => {
    const user = { id: id, name: "Naman" };  // fake server data
    callback(null, user);   // convention: (error, data)
  }, 1000);
}

fetchUser(42, (err, user) => {
  if (err) {
    console.log("Error:", err);
    return;
  }
  console.log("Got user:", user.name);
});

Callback Hell — The Problem with Nesting

When you need to sequence multiple async operations, each depending on the previous result, you have to nest callbacks inside callbacks. This creates the dreaded "pyramid of doom."

Callback Hell — fetch user, then their posts, then comments
fetchUser(1, (err, user) => {
  if (err) return handleError(err);

  fetchPosts(user.id, (err, posts) => {
    if (err) return handleError(err);

    fetchComments(posts[0].id, (err, comments) => {
      if (err) return handleError(err);

      fetchLikes(comments[0].id, (err, likes) => {
        if (err) return handleError(err);

        // Finally have all the data we need
        // But we're 5 levels deep!
        console.log("Likes:", likes);
      });
    });
  });
});

// Problems:
// 1. Hard to read — rightward drift / pyramid shape
// 2. Error handling repeated at every level
// 3. Hard to debug — stack traces don't show useful info
// 4. Hard to refactor or extend

Callbacks vs Promises — The Evolution

Callbacks (old way)

Function passed as argument. Results delivered via callback parameters. Error handling manual at every step. Leads to deeply nested "pyramid" code.

Promises (modern way)

Object representing a future value. Chainable with .then(). Single .catch() handles all errors. Flat, readable code. Covered in Chapter 04.

Try It — Callbacks

Try it
Node.js convention: Async callbacks typically receive (error, result) as their first two arguments. Always check for error first. This pattern is called "error-first callbacks" or "Node-style callbacks."

04

Promises — Basics

A Promise is an object representing the eventual completion (or failure) of an async operation. It's a placeholder for a value you don't have yet.

The Three States of a Promise

⏳ Pending

Initial state. The async operation is in progress. Neither fulfilled nor rejected yet.

✅ Fulfilled

The operation succeeded. The Promise has a value. Your .then() handler runs.

❌ Rejected

The operation failed. The Promise has an error reason. Your .catch() handler runs.

Creating a Promise

new Promise(executor)
// The executor function receives two callbacks:
// resolve(value) — call this when the task succeeds
// reject(reason) — call this when the task fails

const myPromise = new Promise((resolve, reject) => {
  const success = true;

  if (success) {
    resolve("Data loaded!");      // fulfills the promise
  } else {
    reject(new Error("Failed!")); // rejects the promise
  }
});

// Attach handlers with .then() and .catch()
myPromise
  .then(value => {
    console.log("Success:", value);   // "Data loaded!"
  })
  .catch(error => {
    console.log("Error:", error.message);
  });

Simulating Async with setTimeout + Promise

A realistic async Promise pattern
function fetchData(id) {
  return new Promise((resolve, reject) => {
    // Simulate 1-second network delay
    setTimeout(() => {
      if (id > 0) {
        resolve({ id, name: "User " + id });
      } else {
        reject(new Error("Invalid ID: " + id));
      }
    }, 1000);
  });
}

fetchData(5)
  .then(user => console.log("Got user:", user.name))
  .catch(err => console.log("Error:", err.message));

fetchData(-1)
  .then(user => console.log("Got user:", user.name))
  .catch(err => console.log("Error:", err.message));

.finally() — Always Runs

.finally() runs whether the promise fulfills or rejects
fetchData(5)
  .then(user  => console.log("Success:", user.name))
  .catch(err  => console.log("Error:",   err.message))
  .finally(()  => console.log("Cleanup: hide loading spinner"));

// .finally() is perfect for:
// - Hiding loading spinners
// - Closing database connections
// - Re-enabling submit buttons

Try It — Promises

Promises are chainable and their .then() callbacks are queued as microtasks — they run synchronously after the current promise resolves. The sandbox handles these perfectly.

Try it
Always attach .catch(): An unhandled Promise rejection crashes silently in some environments and shows a warning in the browser console. Every Promise chain should end with a .catch() or use a try/catch with async/await (Chapter 07).

05

Promise Chaining

Every .then() returns a new Promise. This lets you chain multiple steps in a flat, readable sequence instead of nesting callbacks.

Returning Values from .then()

Returned values become the next .then()'s input
Promise.resolve(1)
  .then(value => {
    console.log("Step 1:", value);  // 1
    return value + 10;              // pass 11 to next .then()
  })
  .then(value => {
    console.log("Step 2:", value);  // 11
    return value * 2;               // pass 22 to next .then()
  })
  .then(value => {
    console.log("Step 3:", value);  // 22
  });

// Output:
// Step 1: 1
// Step 2: 11
// Step 3: 22

Chaining Async Operations — The Flat Alternative to Callback Hell

Sequential async steps — flat and readable
// Each step returns a new Promise
function getUser(id) {
  return Promise.resolve({ id, name: "Naman" });
}
function getPosts(user) {
  return Promise.resolve([{ id: 1, title: "Hello World", userId: user.id }]);
}
function getComments(post) {
  return Promise.resolve([{ text: "Great post!", postId: post.id }]);
}

// Chain them flat — no nesting!
getUser(1)
  .then(user    => {
    console.log("User:", user.name);
    return getPosts(user);          // return next Promise
  })
  .then(posts   => {
    console.log("Post:", posts[0].title);
    return getComments(posts[0]);   // return next Promise
  })
  .then(comments => {
    console.log("Comment:", comments[0].text);
  })
  .catch(err => {
    console.log("Error anywhere in chain:", err.message);
  });

Error Propagation in Chains

🔑

If any .then() throws an error or returns a rejected Promise, the entire chain skips to the nearest .catch(). You only need one .catch() at the end — not one at every step like with callbacks.

A single .catch() handles errors from any step
Promise.resolve("start")
  .then(v => {
    console.log("Step 1:", v);
    return "step 1 result";
  })
  .then(v => {
    console.log("Step 2:", v);
    throw new Error("Something broke in step 2!");  // triggers catch
  })
  .then(v => {
    console.log("Step 3 — SKIPPED:", v);            // never runs
  })
  .catch(err => {
    console.log("Caught at the end:", err.message);
  });

Try It — Promise Chaining

Try it
Always return: Forget to return inside a .then() and the next step receives undefined. This is one of the most common Promise bugs. If your chain produces undefined, check your return statements.

Promise Combinators

Run multiple Promises simultaneously and handle their collective result. Combinators let you manage parallel async operations elegantly.

Promise.all() — Wait for All

Resolves when ALL promises resolve. Rejects if ANY rejects.
const p1 = Promise.resolve("User data");
const p2 = Promise.resolve("Order data");
const p3 = Promise.resolve("Payment data");

Promise.all([p1, p2, p3])
  .then(([user, order, payment]) => {
    // All three resolved — destructure results in order
    console.log(user);    // "User data"
    console.log(order);   // "Order data"
    console.log(payment); // "Payment data"
  })
  .catch(err => {
    // If ANY of the three rejects, we land here
    console.log("One failed:", err.message);
  });

Promise.allSettled() — Wait for All, Never Rejects

Always resolves with an array of {status, value/reason} objects
const promises = [
  Promise.resolve("Success A"),
  Promise.reject(new Error("Failed B")),
  Promise.resolve("Success C"),
];

Promise.allSettled(promises).then(results => {
  results.forEach((result, i) => {
    if (result.status === "fulfilled") {
      console.log(`[${i}] fulfilled:`, result.value);
    } else {
      console.log(`[${i}] rejected:`, result.reason.message);
    }
  });
});
// Output:
// [0] fulfilled: Success A
// [1] rejected: Failed B
// [2] fulfilled: Success C

Promise.race() — First to Finish Wins

Resolves/rejects as soon as the first Promise settles
// Use case: timeout race — whichever resolves first wins
const fetchData = new Promise(resolve => setTimeout(() => resolve("Data!"), 2000));
const timeout   = new Promise((_, reject) => setTimeout(() => reject(new Error("Timeout!")), 1500));

Promise.race([fetchData, timeout])
  .then(v   => console.log("Winner:", v))
  .catch(err => console.log("Timed out:", err.message));
// Output: Timed out: Timeout!  (1500ms < 2000ms)

Promise.any() — First to Succeed

Resolves as soon as any Promise fulfills. Rejects only if ALL reject.
const p1 = Promise.reject(new Error("Fail A"));
const p2 = Promise.resolve("Success B");
const p3 = Promise.resolve("Success C");

Promise.any([p1, p2, p3])
  .then(first => console.log("First success:", first))  // "Success B"
  .catch(err  => console.log("All failed:", err.message));

// If all reject:
Promise.any([
  Promise.reject(new Error("A")),
  Promise.reject(new Error("B")),
])
  .catch(err => console.log(err.constructor.name, err.message));
// AggregateError All promises were rejected

Comparison Table

MethodResolves when…Rejects when…Use case
Promise.all() All promises fulfill Any one rejects All-or-nothing — load user + cart + preferences together
Promise.allSettled() All promises settle (any state) Never rejects Show partial results — load 10 images, display what worked
Promise.race() First promise settles (any state) First promise rejects Timeout wrapper — reject if response takes too long
Promise.any() First promise fulfills All promises reject Fallback sources — try 3 CDNs, use whichever responds first

Try It — Promise Combinators

Try it
Default choice: Use Promise.all() when all operations must succeed and you need all results. Use Promise.allSettled() when you want partial results and cannot let one failure kill the whole operation.

07

async / await

async/await is syntactic sugar over Promises. It lets you write asynchronous code that reads like synchronous code — no more .then() chains.

The async Keyword

async always returns a Promise
// A regular function
function greet() {
  return "Hello";
}
console.log(greet());               // "Hello"

// An async function — automatically wraps return value in a Promise
async function greetAsync() {
  return "Hello";
}
console.log(greetAsync());          // Promise {fulfilled: "Hello"}
greetAsync().then(v => console.log(v));  // "Hello"

The await Keyword

await pauses the async function until the Promise resolves
function delay(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

async function run() {
  console.log("Start");
  await delay(1000);           // pauses HERE for 1 second
  console.log("After 1s");
  await delay(500);            // pauses HERE for 0.5 seconds
  console.log("After 1.5s total");
}

run();
// Note: code OUTSIDE run() continues immediately
console.log("This logs before 'After 1s'");

Error Handling with try/catch

Replace .catch() with try/catch — much cleaner
async function fetchUser(id) {
  if (id <= 0) throw new Error("Invalid ID");
  return { id, name: "User " + id };
}

async function main() {
  try {
    const user = await fetchUser(5);
    console.log("Got user:", user.name);

    const bad = await fetchUser(-1);  // throws
    console.log("Never reaches here");
  } catch (err) {
    console.log("Caught:", err.message);
  } finally {
    console.log("Always runs");
  }
}

main();

Sequential vs Parallel await

⚡

When you await multiple independent Promises one after another, you wait for them sequentially — total time is the sum of all delays. Use Promise.all() to run them in parallel — total time is the longest single delay.

Sequential vs Parallel — a critical performance difference
function wait(ms, label) {
  return new Promise(res => setTimeout(() => res(label), ms));
}

// SEQUENTIAL — total: 1000 + 800 + 600 = 2400ms
async function sequential() {
  const a = await wait(1000, "A");  // wait 1s
  const b = await wait(800,  "B");  // then wait 0.8s
  const c = await wait(600,  "C");  // then wait 0.6s
  console.log("Sequential:", a, b, c);
}

// PARALLEL — total: max(1000, 800, 600) = 1000ms
async function parallel() {
  const [a, b, c] = await Promise.all([
    wait(1000, "A"),    // all three start simultaneously
    wait(800,  "B"),
    wait(600,  "C"),
  ]);
  console.log("Parallel:", a, b, c);
}

Try It — async/await

Try it
Top-level await: You can only use await inside an async function (or at the top level of an ES module). If you try to use await at the top level of a regular script, you'll get a syntax error. Wrap your code in async function main() { ... } main(); as a workaround.

The Event Loop

The event loop is the mechanism that allows JavaScript to be non-blocking despite being single-threaded. Understanding it explains every "unexpected" async ordering you'll ever see.

The Four Pieces

Call Stack

The stack of currently executing functions. JS runs the function on top, then pops it off. Only one function runs at a time.

Web APIs

Browser-provided features: setTimeout, fetch, DOM events. These run outside the JS engine. When done, they push a callback to a queue.

Microtask Queue

High-priority queue for Promise callbacks (.then(), await). Drained completely before the event loop picks anything from the callback queue.

Callback Queue

Lower-priority queue for timer callbacks (setTimeout, setInterval) and other tasks. Processed one at a time, between renders.

The Event Loop — ASCII Diagram

How JavaScript handles async tasks — the complete picture
┌─────────────────────────────────────────────────────────┐
│                  JavaScript Engine                      │
│                                                         │
│  ┌─────────────────┐    ┌────────────────────────────┐  │
│  │   Call Stack    │    │        Web APIs            │  │
│  │                 │    │  (Browser environment)     │  │
│  │  [ main()     ] │───▶│  ┌──────────────────────┐  │  │
│  │  [ greet()   ] │    │  │ setTimeout(cb, 1000)  │  │  │
│  │  [ console.  ] │    │  │ fetch("https://...")  │  │  │
│  │                 │    │  │ addEventListener(...) │  │  │
│  └────────┬────────┘    │  └──────────┬───────────┘  │  │
│           │             └─────────────│───────────────┘  │
│           │ (stack                    │ (when done,       │
│           │  empty)                   │  push callback)   │
│           │                           ▼                   │
│           │    ┌──────────────────────────────────────┐   │
│           │    │       Microtask Queue (HIGH priority)│   │
│           │    │  Promise.resolve().then(cb)  ◀──────│   │
│           │    │  await (continuation cb)            │   │
│           │    └───────────────────┬──────────────────┘   │
│           │                        │ (drain ALL before     │
│           │                        │  checking below)      │
│           │                        ▼                       │
│           │    ┌──────────────────────────────────────┐   │
│           │    │      Callback Queue (LOW priority)   │   │
│           │    │  setTimeout callback  ◀──────────────│   │
│           │    │  setInterval callback               │   │
│           │    │  DOM event callback                 │   │
│           │    └───────────────────┬──────────────────┘   │
│           │                        │                       │
│           └────────────────────────┘                       │
│               Event Loop: "if stack empty, take            │
│               next task from queue and push to stack"      │
└─────────────────────────────────────────────────────────┘

The Event Loop Rules

Rule 1 — Call Stack First

Synchronous code always runs first. The event loop does nothing until the call stack is completely empty.

Rule 2 — Microtasks Before Tasks

After each task, the entire microtask queue (Promises) is drained before the next callback queue item runs.

Rule 3 — One Task at a Time

The event loop picks one item from the callback queue per "tick." Between ticks, the browser can re-render the page.

Classic Interview Question — What Order?

Predict the output before running it
console.log("1");                           // synchronous

setTimeout(() => console.log("2"), 0);      // callback queue

Promise.resolve().then(() => console.log("3"));  // microtask queue

console.log("4");                           // synchronous

// Answer:
// 1  — synchronous, runs immediately
// 4  — synchronous, runs immediately
// 3  — microtask (Promise), runs after stack clears
// 2  — callback queue (setTimeout), runs after all microtasks

// Why 3 before 2?
// Microtask queue is ALWAYS drained before the callback queue.

Try It — Event Loop Ordering

The sandbox handles microtasks (Promises) perfectly. setTimeout callbacks will fire after the sandbox captures output — watch your F12 console for those. Predict the order before clicking Run.

Try it
Why this matters: Understanding the event loop helps you debug mysterious ordering bugs. When something runs "out of order," it's always because of the distinction between synchronous code, microtasks (Promises), and macrotasks (setTimeout). The order is always: sync → microtasks → macrotasks.

Fetch API

fetch() is the modern browser API for making HTTP requests. It returns a Promise and replaces the older, more verbose XMLHttpRequest.

XMLHttpRequest — The Old Way

XHR — verbose and callback-based (for reference)
// AJAX with XMLHttpRequest (older approach — still works)
const xhr = new XMLHttpRequest();

xhr.open("GET", "https://jsonplaceholder.typicode.com/posts/1");

xhr.onreadystatechange = function () {
  // readyState: 0=UNSENT 1=OPENED 2=HEADERS 3=LOADING 4=DONE
  if (xhr.readyState === 4) {
    if (xhr.status === 200) {
      const data = JSON.parse(xhr.responseText);
      console.log("Title:", data.title);
    } else {
      console.log("Error:", xhr.status, xhr.statusText);
    }
  }
};

xhr.send();

// Common HTTP status codes:
// 200 OK         — success
// 201 Created    — resource created
// 400 Bad Request — client error (bad data)
// 401 Unauthorized — not logged in
// 404 Not Found  — resource doesn't exist
// 500 Server Error — backend crashed

fetch() — The Modern Way

fetch() returns a Promise — two-step: response then body
// Step 1: fetch() returns a Promise for the Response object
// Step 2: .json() parses the body — also returns a Promise

fetch("https://jsonplaceholder.typicode.com/posts/1")
  .then(response => {
    // response is a Response object — NOT the data yet!
    console.log("Status:", response.status);      // 200
    console.log("OK?",    response.ok);           // true if 200-299
    return response.json();  // parse body as JSON — returns a Promise
  })
  .then(data => {
    // data is now the parsed JavaScript object
    console.log("Title:", data.title);
    console.log("Body:",  data.body);
  })
  .catch(err => {
    // Network errors (no internet, DNS failure)
    // NOTE: 404/500 do NOT trigger .catch() — response.ok is false
    console.log("Network error:", err.message);
  });

Checking response.ok — Important!

⚠️

Gotcha: fetch() only rejects on network failure (no internet, DNS error). A 404 or 500 response resolves the Promise — it just sets response.ok to false. Always check response.ok manually.

Always check response.ok for HTTP errors
fetch("https://jsonplaceholder.typicode.com/posts/999999")
  .then(response => {
    if (!response.ok) {
      // Turn HTTP error into a thrown error so .catch() handles it
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
    return response.json();
  })
  .then(data   => console.log("Data:", data))
  .catch(err   => console.log("Error:", err.message));

POST Request with JSON Body

Sending data to a server
const newPost = {
  title:  "My First Post",
  body:   "Hello from JavaScript!",
  userId: 1,
};

fetch("https://jsonplaceholder.typicode.com/posts", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(newPost),   // convert object to JSON string
})
  .then(response => {
    if (!response.ok) throw new Error("HTTP " + response.status);
    return response.json();
  })
  .then(created => {
    console.log("Created with id:", created.id);
    console.log("Title:", created.title);
  })
  .catch(err => console.log("Error:", err.message));

async/await with Fetch

Combining async/await with fetch() produces the cleanest, most readable HTTP request code in JavaScript. This is the pattern you'll use in every real project.

The Cleaner Fetch Pattern

async/await replaces .then() chains — same result, easier to read
// .then() version (Chapter 09)
fetch("https://jsonplaceholder.typicode.com/posts/1")
  .then(res => res.json())
  .then(data => console.log(data.title))
  .catch(err => console.log(err));

// async/await version — reads like synchronous code
async function getPost(id) {
  try {
    const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);

    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }

    const post = await response.json();
    console.log("Title:", post.title);
    return post;
  } catch (err) {
    console.log("Failed:", err.message);
  }
}

A Reusable Fetch Helper

Build a helper you can reuse across your app
async function apiFetch(url, options = {}) {
  const response = await fetch(url, {
    headers: { "Content-Type": "application/json" },
    ...options,
  });

  if (!response.ok) {
    const text = await response.text();
    throw new Error(`HTTP ${response.status}: ${text}`);
  }

  return response.json();
}

// GET
async function main() {
  const post = await apiFetch("https://jsonplaceholder.typicode.com/posts/2");
  console.log("GET:", post.title);

  // POST
  const created = await apiFetch("https://jsonplaceholder.typicode.com/posts", {
    method: "POST",
    body: JSON.stringify({ title: "New Post", userId: 1 }),
  });
  console.log("POST created id:", created.id);
}

main().catch(console.error);

weather-dashboard.js — full solution
const weatherDB = {
  "Mumbai":    { temp: 32, condition: "Humid and Cloudy" },
  "Delhi":     { temp: 38, condition: "Hot and Sunny"   },
  "Chennai":   { temp: 35, condition: "Partly Cloudy"   },
  "Bangalore": { temp: 26, condition: "Pleasant"        },
};

// Returns a Promise that resolves after 1 second with weather data
function getWeather(city) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      const data = weatherDB[city];
      if (data) {
        resolve({ city, ...data });
      } else {
        reject(new Error(`Unknown city: ${city}`));
      }
    }, 1000);
  });
}

// Async function — loads all city weather in parallel
async function loadDashboard(cities) {
  console.log(`Loading weather for: ${cities.join(", ")}...`);
  const startTime = Date.now();

  try {
    // Promise.all: all 3 fetches start simultaneously
    const results = await Promise.all(cities.map(city => getWeather(city)));
    const elapsed = ((Date.now() - startTime) / 1000).toFixed(2);

    console.log(`\n--- Weather Dashboard (loaded in ${elapsed}s) ---`);
    for (const w of results) {
      console.log(`${w.city.padEnd(12)} ${w.temp}°C   ${w.condition}`);
    }
  } catch (err) {
    console.log("Error:", err.message);
  } finally {
    console.log("--- End of report ---");
  }
}

// Bonus: allSettled — partial results even if one city is unknown
async function loadDashboardPartial(cities) {
  console.log("\nLoading with allSettled (partial results on error)...");
  const results = await Promise.allSettled(cities.map(city => getWeather(city)));

  results.forEach(result => {
    if (result.status === "fulfilled") {
      const w = result.value;
      console.log(`${w.city}: ${w.temp}°C — ${w.condition}`);
    } else {
      console.log(`Failed: ${result.reason.message}`);
    }
  });
}
