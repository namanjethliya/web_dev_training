01

Beyond the Browser

The same language you used to make buttons click and forms validate is now running servers, powering mobile apps, building desktop software, and driving machine learning models. JavaScript is everywhere — and that's by design.

One Language, Every Platform

When Brendan Eich created JavaScript in 1995, it was strictly a browser language. In 2009, Ryan Dahl extracted the V8 JavaScript engine from Chrome and ran it as a standalone program called Node.js. That moment changed everything. Suddenly, the same language that ran in a browser could also run a web server, read files from disk, or talk directly to a database.

🌐
Browser
DOM, events, fetch — what you've been learning. Runs on the user's machine.
⚙️
Server
Node.js — handles HTTP requests, reads databases, runs business logic.
📱
Mobile & Desktop
React Native for iOS/Android. Electron for desktop apps like VS Code.

What JavaScript Can Build

Web Apps (Frontend)

React, Vue, Angular — frameworks that manage complex UIs with components, state, and routing.

Servers (Backend)

Node.js + Express — handle HTTP requests, connect to databases, expose REST APIs.

Real-Time Apps

WebSockets + Socket.io — live chat, collaborative editors, live dashboards, multiplayer games.

Desktop Apps

Electron — VS Code, Slack, Figma, and Discord are all built with JavaScript running in Electron.

Mobile Apps

React Native — write once in JavaScript, compile to native iOS and Android apps.

Automation & CLI Tools

Node.js scripts, build tools (Webpack, Vite), test runners (Jest), and linters (ESLint).

The V8 Engine — The Shared Core

⚙️

Whether your JavaScript runs in Chrome or in Node.js, it passes through the V8 engine — Google's open-source JavaScript and WebAssembly engine written in C++. V8 compiles JavaScript to native machine code at runtime, which is why JS is fast enough to run production servers. The same engine, the same language — just a different host environment surrounding it.

What Changes Between Environments

In a Browser

Has access to: window, document, localStorage, fetch, navigator, DOM APIs. Cannot access: the filesystem, OS processes, raw network sockets.

In Node.js

Has access to: filesystem (fs), http module, process, native OS access. Cannot access: window, document, or any DOM — there is no browser here.

The mental model: Think of JavaScript as a universal language and the environment as the toolbox that changes. The core language — variables, functions, loops, arrays, objects, promises — is identical everywhere. Only the available APIs differ.
02

Frontend Frameworks — React

React is the most widely used JavaScript library for building user interfaces. Created by Facebook in 2013, it introduced a component-based model that changed how developers think about building UIs.

Why React Exists — The Problem It Solves

Before React, updating a webpage meant manually reaching into the DOM and changing specific elements. As applications grew in complexity — hundreds of elements, user state, async data — this became unmanageable. A single user action might need to update a notification badge, a list item, a form field, and a sidebar counter simultaneously. Tracking all of that manually was fragile and error-prone.

💡

React's core idea: describe what the UI should look like given the current state — React figures out what needs to change. Instead of saying "find this element and change its text", you say "if isLoggedIn is true, show this; otherwise show that." React re-renders only what changed.

Components — The Building Block

A React application is a tree of components. Each component is a self-contained piece of UI — it manages its own data, renders its own HTML-like output, and handles its own events. You compose components together like Lego bricks.

JSX — React's templating syntax (not runnable in a plain browser)
// A simple React component — a function that returns JSX
function Greeting({ name }) {
  return (
    <div className="card">
      <h2>Hello, {name}!</h2>
      <p>Welcome back.</p>
    </div>
  );
}

// Using the component in another component
function App() {
  return (
    <main>
      <Greeting name="Naman" />
      <Greeting name="Priya" />
      <Greeting name="Rahul" />
    </main>
  );
}

JSX — JavaScript + HTML Syntax

JSX looks like HTML inside JavaScript. It is not valid JavaScript on its own — a build tool (like Vite or Create React App) transpiles it to regular function calls before the browser sees it. You write this:

What you write (JSX)
const element = <h1 className="title">Hello World</h1>;

The build tool compiles it to this:

What the browser actually runs
const element = React.createElement("h1", { className: "title" }, "Hello World");

State — Data That Changes Over Time

State is the dynamic data inside a component. When state changes, React automatically re-renders the component with the new data. The useState hook is how you declare state in a functional component.

Counter component with useState
import { useState } from 'react';

function Counter() {
  // [currentValue, setterFunction] = useState(initialValue)
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
      <button onClick={() => setCount(0)}>
        Reset
      </button>
    </div>
  );
}

// Every time the button is clicked:
// 1. setCount() updates the state
// 2. React re-renders Counter with the new count value
// 3. The UI updates — automatically

The React Ecosystem

React Router

Client-side routing — move between "pages" without refreshing. /home, /profile, /settings all render different components.

State Management

Redux, Zustand, Jotai — libraries for sharing state across many components that don't have a direct parent-child relationship.

Next.js

React framework with server-side rendering, file-based routing, and API routes built in. The standard choice for production React apps.

Component Libraries

Shadcn, Material UI, Chakra UI — pre-built, styled components you can drop into your project. Buttons, modals, forms, tables.

Learning path for React: You need solid JavaScript fundamentals first — this whole course. Then: understand the DOM (Module 4), async/promises (Module 5), and array methods like map() and filter() (Module 3). Then React will make sense. Jumping in without those foundations is the most common source of confusion.

Frontend Frameworks — Angular & Vue

React isn't the only option. Angular and Vue are two other major frameworks, each with a distinct philosophy. Understanding the differences helps you choose the right tool for the job — or evaluate what a team is already using.

Angular — The Full Framework

Angular is a complete, opinionated framework created and maintained by Google. Where React is a library (you bring your own router, state manager, and tooling), Angular ships everything: routing, HTTP client, forms, animations, testing, and a CLI — all in one package.

TypeScript First

Angular is written in TypeScript and expects you to use it. Every component, service, and module is typed. This catches errors at compile time rather than in production.

Dependency Injection

Angular has a built-in dependency injection system — a pattern where services (like an API client) are provided to components automatically, rather than imported manually.

Built for Scale

Angular's strict conventions make it easier to onboard developers onto large codebases. Google, Microsoft, and large enterprises favour it for this reason.

Angular component — app.component.ts (TypeScript)
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',          // used as <app-root> in HTML
  template: `
    <h1>Hello, {{ name }}!</h1>
    <button (click)="greet()">Greet</button>
  `
})
export class AppComponent {
  name: string = 'Naman';        // typed property

  greet(): void {
    alert('Hello, ' + this.name);
  }
}

Vue — The Progressive Framework

Vue was created by Evan You (a former Google engineer who worked on Angular) in 2014. Vue's defining characteristic is that it is progressive — you can add it to an existing HTML page with a single script tag, or build a full SPA with its CLI. You don't have to go all-in from day one.

Gentle Learning Curve

Vue's template syntax is closest to plain HTML. Coming from HTML/CSS, the jump to Vue is smaller than the jump to React or Angular.

Single File Components

A .vue file keeps template, script, and styles together in one file — a pattern many developers find clean and self-contained.

Composition API

Vue 3 introduced the Composition API, inspired by React hooks. It lets you organise logic by feature rather than by lifecycle — making larger components easier to maintain.

Vue 3 Single File Component — Counter.vue
<template>
  <div>
    <p>Count: {{ count }}</p>
    <button @click="increment">+1</button>
  </div>
</template>

<script setup>
import { ref } from 'vue';

const count = ref(0);            // reactive variable

function increment() {
  count.value++;
}
</script>

<style scoped>
button {
  background: #42b883;           /* Vue's signature green */
  color: white;
}
</style>

WebSockets — Real-Time Communication

HTTP was designed for a request-response model: the browser asks, the server answers, the connection closes. WebSockets break this model — they open a persistent, two-way channel so data can flow in both directions at any time, with no polling required.

HTTP vs WebSocket

HTTP Request/Response

Browser sends a request. Server responds. Connection closes. To get new data, the browser must ask again. For live data, you'd have to poll every few seconds — wasteful and slow.

Browser → Server (request)
Server → Browser (response)
Connection closed.

WebSocket Persistent Connection

One handshake to open the connection. Both sides can now send data freely, at any time, in either direction. The connection stays open until either side closes it.

Browser ↔ Server (always open)
Server can push data anytime.
Browser can send anytime.

The ws:// Protocol

🔌

WebSocket connections use the ws:// protocol (or wss:// for secure/encrypted connections, the WebSocket equivalent of HTTPS). The connection starts as an HTTP request with a special Upgrade: websocket header. The server accepts the upgrade, the protocol switches, and from that point on — it's a persistent TCP channel.

What You Can Build with WebSockets

Live Chat Apps

Messages appear on all connected clients instantly, without anyone refreshing. WhatsApp Web uses WebSockets.

Live Dashboards

Stock price tickers, server monitoring panels, sports scoreboards — data pushed from server to all clients the moment it changes.

Collaborative Tools

Google Docs-style real-time editing — changes from one user propagate to all other users in milliseconds.

Multiplayer Games

Browser-based games need sub-100ms updates. WebSockets make this possible by eliminating per-request overhead.

WebSocket Client Code — Browser Side

The browser's built-in WebSocket API is straightforward. You open a connection, listen for events, and send/receive messages.

WebSocket client — browser JavaScript
// 1. Open a connection to the WebSocket server
const socket = new WebSocket('wss://example.com/chat');

// 2. Connection opened — fire and forget messages
socket.addEventListener('open', (event) => {
  console.log('Connected to server');
  socket.send('Hello Server!');           // send a string
  socket.send(JSON.stringify({            // or send JSON
    type: 'join',
    room: 'general',
    user: 'Naman'
  }));
});

// 3. Receive messages from the server
socket.addEventListener('message', (event) => {
  const data = JSON.parse(event.data);    // parse incoming JSON
  console.log('Message from server:', data);
  // update the UI with the new message
});

// 4. Handle errors
socket.addEventListener('error', (event) => {
  console.error('WebSocket error:', event);
});

// 5. Connection closed
socket.addEventListener('close', (event) => {
  console.log('Disconnected. Code:', event.code);
});

// 6. Close from the client side
// socket.close();

Socket.io — WebSockets Made Easy

The raw WebSocket API works, but managing rooms, reconnections, broadcasting, and fallbacks is complex. Socket.io is a popular library that wraps WebSockets and adds these features out of the box.

What Socket.io adds

Automatic reconnection on disconnect. Named events instead of raw strings. Rooms and namespaces for grouping connections. Fallback to HTTP long-polling for environments where WebSockets are blocked. Works in the browser and Node.js.

Socket.io syntax
// Client
const socket = io('http://localhost:3000');
socket.emit('chat', { text: 'Hello' });
socket.on('chat', (msg) => {
  console.log(msg.text);
});

What is Node.js?

Node.js is a JavaScript runtime built on Chrome's V8 engine. It lets you run JavaScript outside of a browser — on your machine, on a server, anywhere. Same language, completely different superpowers.

How Node.js Works

🔧

Node.js takes the V8 engine (which only understands JavaScript) and wraps it with a C++ layer that provides access to the operating system: the filesystem, network sockets, timers, processes, and more. It also implements the event loop — a mechanism that handles thousands of concurrent operations without blocking, using non-blocking I/O.

Event-Driven, Non-Blocking I/O

This is the phrase you'll see in every Node.js description. It means Node doesn't sit and wait for slow operations (reading a file, making a network request). It kicks off the operation, registers a callback, and moves on to handle other work. When the operation completes, the callback runs.

Traditional (Blocking)

Thread 1: Read file... (waiting) ... (waiting) ... done. Only then can it handle the next request. You need one thread per concurrent user.

Node.js (Non-Blocking)

Read file → register callback → handle next request → handle next → ... → file ready → callback runs. One thread handles thousands of concurrent operations.

Node.js — Creating an HTTP Server

Node.js ships with a built-in http module. With it, you can create a web server that listens for incoming requests and sends back responses — no Apache, no Nginx required.

🖥️

The code in this chapter requires Node.js installed on your machine — it cannot run in a browser. Save the code to a file (e.g. server.js) and run it from your terminal with node server.js. Then open http://localhost:3000 in your browser.

A Minimal HTTP Server

Run in Terminal server.js → node server.js 
// server.js — run with: node server.js
const http = require('http');      // 1. import Node's built-in http module

const PORT = 3000;

// 2. Create the server — callback runs on every request
const server = http.createServer((req, res) => {

  // 3. Log what the client asked for
  console.log(`${req.method} ${req.url}`);

  // 4. Set the response status and headers
  res.writeHead(200, { 'Content-Type': 'text/plain' });

  // 5. Send the response body and close the connection
  res.end('Hello from Node.js!\n');
});

// 6. Start listening
server.listen(PORT, () => {
  console.log(`Server running at http://localhost:${PORT}`);
});
Terminal Output what you see after running node server.js 
$ node server.js
Server running at http://localhost:3000
GET /             ← logged when you visit localhost:3000
GET /favicon.ico  ← browser automatically requests the favicon

Routing — Handling Different URLs

By inspecting req.url and req.method, you can respond differently to different requests — this is routing.

Run in Terminal server-routes.js → node server-routes.js 
// server-routes.js
const http = require('http');

const server = http.createServer((req, res) => {
  const { url, method } = req;

  // Route: GET /
  if (url === '/' && method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'text/html' });
    res.end('<h1>Home Page</h1>');

  // Route: GET /about
  } else if (url === '/about' && method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ name: 'My App', version: '1.0' }));

  // 404 for everything else
  } else {
    res.writeHead(404, { 'Content-Type': 'text/plain' });
    res.end('404 Not Found');
  }
});

server.listen(3000, () => console.log('Server on port 3000'));

The Request and Response Objects

req — IncomingMessage

req.url — the requested path (/about)
req.method — HTTP verb (GET, POST)
req.headers — all HTTP headers as an object
req.on('data', cb) — stream the request body

res — ServerResponse

res.writeHead(status, headers) — set status + headers
res.setHeader(name, value) — set a single header
res.write(chunk) — send partial body
res.end(data) — send final body and close

Why You'll Use Express Instead

The raw http module works, but manually parsing URLs, handling query strings, parsing JSON bodies, and setting correct headers for every route gets tedious fast. Express is a minimal framework that adds exactly this convenience layer — covered in Chapter 07.

Anatomy of a server: Every web server does the same three things — (1) listen on a port, (2) receive a request, (3) send a response. Whether it's Node's raw http, Express, Fastify, Koa, or any other framework — they all implement these three steps. Once you understand the raw version, every framework becomes easy to read.

Node.js — File System & Express

Two of the most-used capabilities in Node.js: reading and writing files with the fs module, and building web servers far more elegantly with Express.js.

The fs Module — Reading Files

The fs (file system) module gives Node.js access to the host machine's filesystem. You can read, write, delete, rename, and watch files and directories.

Run in Terminal read-file.js → node read-file.js 
// read-file.js — requires a file called "data.txt" to exist
const fs = require('fs');

// Async (non-blocking) — preferred
fs.readFile('data.txt', 'utf8', (err, data) => {
  if (err) {
    console.error('Error reading file:', err.message);
    return;
  }
  console.log('File contents:', data);
});

console.log('This runs BEFORE the file is read!');
// Node doesn't wait — it reads asynchronously and runs the callback later

// Sync (blocking) — use only in scripts, never in a server
try {
  const contents = fs.readFileSync('data.txt', 'utf8');
  console.log('Sync read:', contents);
} catch (err) {
  console.error(err.message);
}

Writing Files

Run in Terminal write-file.js → node write-file.js 
// write-file.js
const fs = require('fs');

// Write (overwrites the file if it exists)
fs.writeFile('output.txt', 'Hello from Node.js!\n', (err) => {
  if (err) throw err;
  console.log('File written successfully');
});

// Append (adds to end without overwriting)
fs.appendFile('output.txt', 'Another line.\n', (err) => {
  if (err) throw err;
  console.log('Line appended');
});

// Modern async/await style with fs.promises
const fsPromises = require('fs').promises;

async function writeData() {
  await fsPromises.writeFile('result.json', JSON.stringify({ ok: true }));
  const data = await fsPromises.readFile('result.json', 'utf8');
  console.log('Read back:', data);
}

writeData();

require vs import — Two Module Systems

CommonJS — require (classic Node)
const fs  = require('fs');
const path = require('path');
const express = require('express');

// Export your own code:
module.exports = { myFunction };

Default in Node.js. File extension: .js

ES Modules — import (modern)
import fs from 'fs';
import path from 'path';
import express from 'express';

// Export your own code:
export { myFunction };

Use "type": "module" in package.json or .mjs extension.

Express.js — The Most Popular Node Framework

Express is a minimal, unopinionated Node.js framework. It wraps the raw http module and adds routing, middleware, easier request/response handling, and a plugin ecosystem. It's been the de facto Node.js web framework for over a decade.

Terminal set up Express in a new project 
$ mkdir express-app && cd express-app
$ npm init -y
$ npm install express
Run in Terminal app.js → node app.js 
// app.js — a complete Express server
const express = require('express');
const app = express();

// Middleware: parse incoming JSON request bodies
app.use(express.json());

// Route: GET /
app.get('/', (req, res) => {
  res.send('Hello from Express!');
});

// Route: GET /users
app.get('/users', (req, res) => {
  const users = [
    { id: 1, name: 'Naman' },
    { id: 2, name: 'Priya' },
  ];
  res.json(users);            // automatically sets Content-Type: application/json
});

// Route: POST /users — receive JSON body
app.post('/users', (req, res) => {
  const { name } = req.body;
  console.log('New user:', name);
  res.status(201).json({ id: 3, name });
});

// Route params: GET /users/42
app.get('/users/:id', (req, res) => {
  const { id } = req.params;
  res.json({ id: Number(id), name: 'User ' + id });
});

// 404 handler — must be last
app.use((req, res) => {
  res.status(404).json({ error: 'Not found' });
});

app.listen(3000, () => {
  console.log('Express server running on http://localhost:3000');
});

What Express Adds Over Raw http

Clean Routing

app.get('/users/:id', handler) instead of a chain of if/else on req.url. Route parameters extracted automatically.

Middleware

Functions that run before your route handler — for logging, authentication, JSON parsing, CORS headers. Stack them with app.use().

res.json()

Serialises an object to JSON and sets the correct Content-Type header automatically. One line instead of four.

Ecosystem

Thousands of Express middleware packages: Passport (auth), Multer (file upload), Helmet (security), morgan (logging), CORS, rate limiting.
