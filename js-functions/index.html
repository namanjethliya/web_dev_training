01

What are Functions?

A function is a named, reusable block of code that performs a specific task. Instead of writing the same code multiple times, you write it once inside a function and call it whenever you need it.

The Problem Functions Solve

Imagine you're building a quiz app and you need to calculate a grade from a score in five different places. Without functions, you'd copy the same if/else block every time. If the grading rule changes, you'd have to find and update all five copies — and probably miss one.

Without Functions (BAD)

Write the same logic in 5 places. Change the rule → find and update 5 spots. Miss one → bug. This violates the DRY principle.

With Functions (GOOD)

Write the logic once. Call it 5 times. Change the rule → update 1 place. Everything stays in sync automatically.

The DRY Principle

💡

DRY = Don't Repeat Yourself. If you find yourself writing the same code more than once, it belongs in a function. Functions are the primary tool for eliminating repetition and making code maintainable. Every time you call a function instead of duplicating code, you're being a better programmer.

What Functions Give You

Reusability

Write once, call anywhere. The same calculateGrade() function works for every student score, every time.

Readability

A function named sendEmail() tells you exactly what that line does — you don't need to read 20 lines of networking code to understand it.

Testability

You can test a function in isolation. If add(2, 3) returns 5, that function works — no matter where in your app it's called.

Abstraction

Functions let you think at a higher level. processOrder() hides complexity. You call it without needing to know how it works internally.

The Anatomy of a Function

The three parts of every function
//  ┌── keyword   ┌── name      ┌── parameters (inputs)
    function      greet        (name) {
      return "Hello, " + name + "!";
    //  ↑ return value (output)
    }

    // Calling (invoking) the function:
    const message = greet("Naman");
    console.log(message); // "Hello, Naman!"

    // Call it again with different input:
    console.log(greet("Priya")); // "Hello, Priya!"
    console.log(greet("Rahul")); // "Hello, Rahul!"
Defining vs Calling: Writing function greet(name) { ... } defines the function — it doesn't run yet. Writing greet("Naman") calls it — that's when the code inside actually executes. A function can be defined once but called thousands of times.
02

Function Declaration vs Expression

There are two primary ways to define a function in JavaScript. They look similar but behave differently in one crucial way: hoisting.

Function Declaration

The classic syntax. Uses the function keyword followed by a name, then the function body.

Function Declaration syntax
// Declaration
function add(a, b) {
  return a + b;
}

// Usage
console.log(add(3, 4)); // 7
console.log(add(10, 20)); // 30

Function Expression

A function stored in a variable. The function itself has no name (it's anonymous) — the variable holds a reference to it.

Function Expression syntax
// Expression — assigned to a variable
const multiply = function(a, b) {
  return a * b;
};

// Usage — same as a declaration
console.log(multiply(3, 4)); // 12
console.log(multiply(5, 5)); // 25

The Key Difference: Hoisting

⚠️

Hoisting means JavaScript moves declarations to the top of their scope before executing code. Function declarations are fully hoisted — you can call them before the line where they're written. Function expressions are NOT hoisted — you must define them before calling them.

Hoisting in action
// ✓ Function Declaration — can call BEFORE the definition
console.log(greet("World")); // "Hello, World!" — works!

function greet(name) {
  return "Hello, " + name + "!";
}

// ✗ Function Expression — CANNOT call before the definition
console.log(square(4)); // ReferenceError: Cannot access 'square' before initialization

const square = function(n) {
  return n * n;
};

Side by Side Comparison

FeatureFunction DeclarationFunction Expression
Syntax function name() {} const name = function() {}
Hoisted? Yes — usable before the definition No — must define before calling
Has a name? Yes — always has a name Usually anonymous (no name)
When to use General-purpose named functions When assigning to variables, passing as arguments

IIFE — Immediately Invoked Function Expression

An IIFE is a function expression that runs immediately after it's defined. Useful for creating a private scope and avoiding polluting the global namespace.

IIFE pattern
// Syntax: wrap in () and immediately call with ()
(function() {
  const secret = "only accessible here";
  console.log("IIFE ran!", secret);
})();

// With a parameter
(function(name) {
  console.log("Hello from IIFE,", name);
})("Naman");

// Arrow function IIFE
(() => {
  console.log("Arrow IIFE ran!");
})();

03

Parameters & Return Values

Parameters are the inputs a function accepts. Return values are the outputs it sends back. Together they define a function's contract: what it needs and what it produces.

Parameters vs Arguments

Parameters

The names listed in the function definition. They are placeholders — like variables that will receive values when the function is called.

function add(a, b) { }
Arguments

The actual values passed when calling the function. They are assigned to the parameters in the same order.

add(3, 5); // a=3, b=5

Multiple Parameters

app.js — multiple parameters in action
// Function with multiple parameters
function describeTriangle(base, height, unit) {
  const area = 0.5 * base * height;
  return "Triangle: base=" + base + unit + ", height=" + height + unit + ", area=" + area + unit + "²";
}

console.log(describeTriangle(5, 8, "cm"));
// "Triangle: base=5cm, height=8cm, area=20cm²"

// Too few arguments — missing param becomes undefined
function greet(firstName, lastName) {
  return "Hello, " + firstName + " " + lastName;
}
console.log(greet("Naman"));
// "Hello, Naman undefined" — lastName is undefined

// Too many arguments — extras are silently ignored
console.log(greet("Naman", "Jethliya", "extra", "ignored"));
// "Hello, Naman Jethliya" — extra args are ignored

Return Values

The return statement sends a value back to the caller and immediately stops the function. Without return, a function returns undefined.

return — sending a value back
// Function with a return value
function celsiusToFahrenheit(c) {
  return (c * 9/5) + 32;
}

const boiling = celsiusToFahrenheit(100);
console.log("100°C =", boiling, "°F"); // "100°C = 212 °F"

// The returned value can be used in expressions
console.log("Body temp:", celsiusToFahrenheit(37).toFixed(1), "°F"); // "98.6 °F"

// Without return — returns undefined
function sayHello(name) {
  console.log("Hello, " + name + "!");
  // No return statement
}
const result = sayHello("Naman"); // logs "Hello, Naman!"
console.log("Return value:", result); // "Return value: undefined"

Early Return — Guard Clauses

You can return early from a function to handle invalid inputs or edge cases before the main logic runs. This keeps code flat and readable.

Early return pattern
function divide(a, b) {
  // Guard clause — handle edge case first
  if (b === 0) {
    return "Error: Cannot divide by zero";
  }
  // Main logic — only reached if b !== 0
  return a / b;
}

console.log(divide(10, 2));  // 5
console.log(divide(10, 0));  // "Error: Cannot divide by zero"

// Without early return (harder to read)
function divideHarder(a, b) {
  let result;
  if (b !== 0) {
    result = a / b;
  } else {
    result = "Error: Cannot divide by zero";
  }
  return result;  // same outcome, more nesting
}

Try It — Parameters & Return

Try it
Return early, return often: Use early returns (guard clauses) to handle invalid inputs at the top of your function. This avoids deeply nested if/else and keeps the "happy path" code unindented and easy to read.

04

Arrow Functions

Arrow functions are a shorter syntax for writing function expressions, introduced in ES6. They're everywhere in modern JavaScript and are especially convenient for short, single-purpose functions.

The Syntax Progression

There are four forms of arrow function syntax, depending on how many parameters you have and whether you need a function body.

Arrow function syntax — from most to least verbose
// 1. Full arrow function (multiple params, multi-line body)
const add = (a, b) => {
  const result = a + b;
  return result;
};

// 2. Multiple params, single expression — omit braces & return
const multiply = (a, b) => a * b;

// 3. Single param — can omit the parentheses around it
const double = n => n * 2;

// 4. No params — must keep empty parentheses
const greet = () => "Hello, World!";

console.log(add(3, 4));       // 7
console.log(multiply(3, 4));  // 12
console.log(double(7));       // 14
console.log(greet());         // "Hello, World!"

Implicit Return

💡

When you omit the curly braces {} from an arrow function, the expression after => is automatically returned. This is called an implicit return. It only works for a single expression — if you need multiple lines, you must use curly braces and an explicit return.

Explicit vs implicit return
// Explicit return (with braces)
const squareExplicit = (n) => {
  return n * n;
};

// Implicit return (without braces) — same result, shorter
const squareImplicit = n => n * n;

console.log(squareExplicit(5)); // 25
console.log(squareImplicit(5)); // 25

// Returning an object literal — must wrap in extra ()
// Otherwise JS thinks the {} are the function body!
const makeUser = (name, age) => ({ name: name, age: age });
console.log(makeUser("Naman", 25)); // { name: "Naman", age: 25 }

Arrow Functions vs Regular Functions

FeatureRegular FunctionArrow Function
Syntax function(a) { return a; } a => a
Implicit return No — always need return Yes — when no {}
Hoisted? Yes (declarations only) No
this binding Has its own this Inherits this from surrounding scope
Best used for Methods, constructors, complex logic Short callbacks, array methods, simple transforms

Try It — Arrow Functions

Try it
Style guide: Use arrow functions for short callbacks and transformations. Use regular function declarations for top-level, named functions that you want to be self-documenting and hoistable. Both are equally valid — consistency matters most.

Default Parameters

Default parameters let you specify a fallback value for a parameter that is not provided when the function is called. They eliminate manual undefined checks and make function calls more flexible.

The Problem They Solve

Before default parameters (ES5 way — verbose)
// The old way — manual check inside the function
function greet(name) {
  name = name !== undefined ? name : "stranger";
  return "Hello, " + name + "!";
}

// The even older way — using ||
function greetOld(name) {
  name = name || "stranger";  // works but has a bug: 0 or "" also trigger it
  return "Hello, " + name + "!";
}
With default parameters (ES6+ — clean)
// Default parameter — assigned in the signature
function greet(name = "stranger") {
  return "Hello, " + name + "!";
}

console.log(greet("Naman"));  // "Hello, Naman!"
console.log(greet());         // "Hello, stranger!"
console.log(greet(undefined)); // "Hello, stranger!" — undefined triggers the default
console.log(greet(null));     // "Hello, null!"     — null does NOT trigger the default

Multiple Defaults and Expressions

Defaults can be any expression
// Multiple default parameters
function createProfile(name = "Anonymous", age = 0, role = "viewer") {
  return `${name} (${age}) — ${role}`;
}

console.log(createProfile("Naman", 25, "admin")); // "Naman (25) — admin"
console.log(createProfile("Priya", 22));           // "Priya (22) — viewer"
console.log(createProfile());                      // "Anonymous (0) — viewer"

// Defaults can use previous parameters
function range(start, end = start + 10) {
  return `${start} to ${end}`;
}

console.log(range(5, 15)); // "5 to 15"
console.log(range(5));     // "5 to 15" — end defaults to 5+10

// Default can call a function
function getDefaultDate() {
  return new Date().getFullYear();
}

function event(title, year = getDefaultDate()) {
  return `${title} (${year})`;
}

console.log(event("Tech Summit", 2023)); // "Tech Summit (2023)"
console.log(event("Hackathon"));         // "Hackathon (current year)"

Try It — Default Parameters

Try it
null vs undefined: Only undefined (or an omitted argument) triggers a default parameter. Passing null explicitly does NOT — the parameter will be null. This matters when you're explicitly signalling "no value" vs "caller didn't provide one".

Rest Parameters

The rest parameter syntax (...name) lets a function accept any number of arguments and collects them into a real array. It's the modern replacement for the old arguments object.

The Syntax

Rest parameter — collects remaining args into an array
// ...nums collects all arguments into an array
function sum(...nums) {
  console.log(nums); // [1, 2, 3, 4, 5] — a real Array
  let total = 0;
  for (const n of nums) {
    total += n;
  }
  return total;
}

console.log(sum(1, 2, 3));          // 6
console.log(sum(10, 20, 30, 40));   // 100
console.log(sum(5));                 // 5
console.log(sum());                  // 0 — empty array, loop doesn't run

Rest with Named Parameters

The rest parameter must always be last. Named parameters before it capture specific arguments; the rest collect everything remaining.

Named params + rest — powerful combination
// First arg is the operation label, rest are the numbers
function calculate(operation, ...numbers) {
  const total = numbers.reduce((acc, n) => acc + n, 0);
  return `${operation}: ${numbers.join(" + ")} = ${total}`;
}

console.log(calculate("Sum", 1, 2, 3));        // "Sum: 1 + 2 + 3 = 6"
console.log(calculate("Total", 10, 20, 30));   // "Total: 10 + 20 + 30 = 60"

// Building a tag template
function tag(label, ...items) {
  return `[${label}] ${items.join(", ")}`;
}

console.log(tag("Fruits", "apple", "banana", "mango"));
// "[Fruits] apple, banana, mango"

// ✗ Rest must be the LAST parameter
// function bad(...items, extra) { } // SyntaxError!

Rest vs the Old arguments Object

arguments (old way)

Available in regular functions. Array-like but NOT a real array — you can't call .map() or .filter() on it. Doesn't work in arrow functions.

...rest (modern)

A real Array. Works with all array methods. Works in arrow functions. Clearly shows intent. Always prefer this.

Try It — Rest Parameters

Try it
When to use rest: Reach for rest parameters when you want a function to accept a variable number of arguments — like a sum(), a logger, or a tag builder. It's especially useful in utility functions that operate on collections of values.

Spread Syntax

The spread operator (...) does the opposite of rest: it expands an iterable (array, string) into individual elements. Same three dots, completely different purpose depending on where you use them.

Spreading into Function Calls

Pass an array as individual arguments
const numbers = [3, 1, 7, 2, 9, 4];

// Without spread — Math.max can't take an array
console.log(Math.max(numbers));   // NaN — wrong!

// With spread — expands the array into individual args
console.log(Math.max(...numbers));  // 9 — correct!
console.log(Math.min(...numbers));  // 1

// Same with a custom function
function add(a, b, c) {
  return a + b + c;
}

const vals = [5, 10, 15];
console.log(add(...vals)); // 30 — equivalent to add(5, 10, 15)

Spreading into Array Literals

Combine, copy, and extend arrays
const fruits  = ["apple", "banana"];
const veggies = ["carrot", "spinach"];

// Combine arrays
const food = [...fruits, ...veggies];
console.log(food); // ["apple", "banana", "carrot", "spinach"]

// Copy an array (shallow copy)
const fruitsCopy = [...fruits];
fruitsCopy.push("mango");
console.log(fruits);     // ["apple", "banana"] — original unchanged
console.log(fruitsCopy); // ["apple", "banana", "mango"]

// Insert elements in the middle
const nums = [1, 2, 5, 6];
const full = [1, 2, 3, 4, ...nums.slice(2)]; // creative, but just an example
const withMiddle = [...nums.slice(0, 2), 3, 4, ...nums.slice(2)];
console.log(withMiddle); // [1, 2, 3, 4, 5, 6]

Spread with Objects

Copy and merge objects with spread
const base = { theme: "dark", lang: "en" };
const extra = { fontSize: 16, lang: "hi" }; // note: lang will be overwritten

// Merge objects — later properties override earlier ones
const settings = { ...base, ...extra };
console.log(settings);
// { theme: "dark", lang: "hi", fontSize: 16 }

// Create an updated copy (immutable update pattern)
const user = { name: "Naman", age: 25, city: "Mumbai" };
const updatedUser = { ...user, age: 26 }; // only age changes
console.log(updatedUser);
// { name: "Naman", age: 26, city: "Mumbai" }
console.log(user); // { name: "Naman", age: 25, city: "Mumbai" } — unchanged

Rest vs Spread — Same Symbol, Opposite Direction

Rest (...) — collects into an array

Used in function parameters. Gathers individual arguments into one array.

function fn(...args) { }
Spread (...) — expands out of an array

Used in function calls or array/object literals. Expands one array into individual values.

fn(...arr)

Try It — Spread Syntax

Try it
Immutable updates: The object spread pattern ({ ...original, field: newValue }) is one of the most important patterns in React and modern JS. It creates a new object with one field changed, without mutating the original — a cornerstone of predictable state management.

Scope

Scope determines where in your code a variable is accessible. Understanding scope prevents "variable not defined" errors and helps you write code where variables exist only as long as they're needed.

The Three Levels of Scope

Global Scope

Variables declared outside any function or block. Accessible everywhere in your script. Avoid polluting global scope — it leads to naming conflicts.

Function Scope

Variables declared inside a function with var, let, or const. Only accessible inside that function.

Block Scope

Variables declared with let or const inside {} (if, for, while). Only accessible within that block.

Global, function, and block scope in action
// Global scope — accessible everywhere
const appName = "MyApp";

function showInfo() {
  // Function scope — only accessible inside showInfo
  const version = "1.0";
  console.log(appName + " v" + version); // can access global

  if (true) {
    // Block scope — only accessible inside this if block
    const message = "Feature enabled";
    let counter = 0;
    console.log(message); // works here
  }

  // console.log(message); // ✗ ReferenceError — block scope!
  // console.log(counter); // ✗ ReferenceError — block scope!
}

showInfo();
console.log(appName); // ✓ — global
// console.log(version); // ✗ ReferenceError — function scope!

var vs let — The Critical Scope Difference

⚠️

var is function-scoped — it leaks out of blocks like if and for. let and const are block-scoped — they stay contained within {}. This is one of the main reasons modern JavaScript uses let/const over var.

var leaks out of blocks — let does not
// var leaks out of the for block
for (var i = 0; i < 3; i++) {
  // i is accessible here
}
console.log(i); // 3 — ✗ var leaked out of the for block!

// let stays inside the block
for (let j = 0; j < 3; j++) {
  // j is accessible here
}
// console.log(j); // ✗ ReferenceError — let did NOT leak

// Classic bug with var in loops
for (var k = 0; k < 3; k++) {
  setTimeout(() => console.log("var k:", k), 0);
}
// Logs: "var k: 3" three times — k was shared!

for (let m = 0; m < 3; m++) {
  setTimeout(() => console.log("let m:", m), 0);
}
// Logs: "let m: 0", "let m: 1", "let m: 2" — each iteration gets its own m

Lexical Scoping

JavaScript uses lexical scoping — a function can access variables from the scope where it was defined, not where it is called. Inner functions can always see outer variables.

Lexical scope — inner can see outer
const language = "JavaScript";  // outer scope

function showLanguage() {
  const version = "ES2024";       // middle scope

  function displayFull() {        // inner function
    // Can see both outer and middle scope variables
    console.log(language + " " + version); // "JavaScript ES2024"
  }

  displayFull();
}

showLanguage();

The window Object

In a browser, global variables become properties of the window object — the global scope. This is why naming conflicts are dangerous.

Global scope and the window object
var globalVar = "I am global";
console.log(window.globalVar); // "I am global" — attached to window

// This is why top-level var declarations can conflict:
// if two scripts both declare var name = "..." — they overwrite each other

// const and let at the top level are NOT on window
const myConst = "not on window";
console.log(window.myConst); // undefined

Try It — Scope

Try it
Scope hierarchy: Always use the tightest scope possible. Declare variables inside the block or function that needs them. Variables with smaller scope are easier to reason about — you know exactly where they exist and where they can be changed.

Closures

A closure is a function that remembers the variables from the scope where it was created — even after that outer scope has finished executing. Closures are one of JavaScript's most powerful and frequently used features.

The Concept

🔒

When a function is defined inside another function, the inner function "closes over" the outer function's variables. Those variables are captured and kept alive in memory as long as the inner function exists — even after the outer function returns. The inner function plus its captured environment is called a closure.

The Classic Counter Example

Counter closure — private state
function makeCounter() {
  let count = 0; // this variable is "closed over"

  return function() {
    count++;        // accesses count from outer scope
    return count;
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3

// Each call to makeCounter creates a NEW, independent closure
const counter2 = makeCounter();
console.log(counter2()); // 1 — starts fresh!
console.log(counter());  // 4 — counter1 continues where it left off

Why is count not reset each time?

Normally, when a function finishes, its local variables are garbage-collected. But when an inner function holds a reference to count, JavaScript keeps count alive. The inner function has a private, persistent variable that only it can access.

Closure for data privacy — encapsulation
function makeBankAccount(initialBalance) {
  let balance = initialBalance; // private — cannot be accessed directly

  return {
    deposit:  amount  => { balance += amount; return balance; },
    withdraw: amount  => {
      if (amount > balance) return "Insufficient funds";
      balance -= amount;
      return balance;
    },
    getBalance: () => balance
  };
}

const account = makeBankAccount(1000);
console.log(account.getBalance());   // 1000
console.log(account.deposit(500));   // 1500
console.log(account.withdraw(200));  // 1300
console.log(account.withdraw(5000)); // "Insufficient funds"
// account.balance — undefined! The variable is private via closure.

Closure for Function Factories

Creating customized functions with closures
// A factory that creates multiplier functions
function makeMultiplier(factor) {
  return n => n * factor; // factor is closed over
}

const double = makeMultiplier(2);
const triple = makeMultiplier(3);
const tenX   = makeMultiplier(10);

console.log(double(5));  // 10
console.log(triple(5));  // 15
console.log(tenX(7));    // 70

// Each function remembers its own factor
console.log(double(100)); // 200
console.log(triple(100)); // 300

Try It — Closures

Try it
Closures everywhere: Every time you write an inner function that uses a variable from the outer function, you're using a closure. Array callbacks (.map(n => n * factor)), event handlers, and setTimeout callbacks are all closures in practice.

Callbacks

A callback is a function passed as an argument to another function, to be called later. In JavaScript, functions are values — you can pass them around just like numbers or strings. Callbacks are the foundation of asynchronous JavaScript.

Functions are First-Class Values

💡

In JavaScript, functions are "first-class citizens" — they can be stored in variables, passed as arguments to other functions, and returned from functions. This is what makes callbacks possible, and it's one of the things that makes JavaScript so flexible.

Passing a function as an argument
// A function that accepts a callback
function doTwice(action) {
  action(); // call the passed-in function
  action();
}

// Define a function
function sayHello() {
  console.log("Hello!");
}

// Pass it as a callback (notice: no parentheses — we pass the function itself)
doTwice(sayHello); // "Hello!" "Hello!"

// Inline arrow function callback — same result, shorter
doTwice(() => console.log("World!")); // "World!" "World!"

Callbacks with Data

Passing data to and from callbacks
// A function that processes each item and calls a callback with the result
function processEach(items, callback) {
  for (const item of items) {
    const result = callback(item);
    console.log(result);
  }
}

// Named function callback
function formatUpper(word) {
  return word.toUpperCase();
}

const words = ["apple", "banana", "mango"];
processEach(words, formatUpper);
// "APPLE"
// "BANANA"
// "MANGO"

// Arrow function callback — inline
processEach(words, word => "Fruit: " + word);
// "Fruit: apple" etc.

Built-in Callbacks: Array Methods

The most common use of callbacks in everyday JavaScript is with array methods. Every call to .map(), .filter(), .forEach(), and .reduce() takes a callback.

Callbacks with array methods
const students = [
  { name: "Naman",  score: 88 },
  { name: "Priya",  score: 95 },
  { name: "Rahul",  score: 72 },
  { name: "Anjali", score: 60 },
];

// .map() — transform each element
const names = students.map(s => s.name);
console.log(names); // ["Naman", "Priya", "Rahul", "Anjali"]

// .filter() — keep only matching elements
const topStudents = students.filter(s => s.score >= 85);
console.log(topStudents.map(s => s.name)); // ["Naman", "Priya"]

// .forEach() — run a callback for each element
students.forEach(s => {
  const grade = s.score >= 90 ? "A" : s.score >= 80 ? "B" : "C";
  console.log(`${s.name}: ${grade}`);
});

Practical Uses of Callbacks

Event handling

When a button is clicked, your callback runs. The browser calls it when the event happens.

btn.addEventListener("click", () => alert("clicked!"))
Timers

Schedule code to run after a delay. The timer calls your callback when the time is up.

setTimeout(() => console.log("done"), 1000)
Array methods

Transform, filter, and reduce arrays. The method calls your callback for each element.

[1,2,3].map(n => n * 2)

Ask Yourself Before Starting

Q1

How does calculate(a, op, b) know which math operation to run? It accepts op as a parameter — but what type is op?

Hint: op is a function. You pass add, subtract, etc. as the argument. Inside calculate, you call op(a, b). This is the callback pattern from Ch.10.
Q2

In makeLogger(prefix), the returned function uses prefix. But makeLogger has already returned by then. Why does prefix still exist?

Hint: The inner function closes over prefix. Closures (Ch.09) keep outer variables alive as long as the inner function exists.
Q3

How does sumAll(1, 2, 3, 4, 5) work when the number of arguments isn't fixed?

Hint: Use rest parameters (...nums). Inside the function, nums is an array. Use .reduce() to add all elements.

function-toolkit.js — full solution
// 1. Arrow math functions
const add      = (a, b) => a + b;
const subtract = (a, b) => a - b;
const multiply = (a, b) => a * b;
const divide   = (a, b) => {
  if (b === 0) return "Error: division by zero";
  return a / b;
};

// 2. Dispatcher — callback pattern
function calculate(a, op, b) {
  return op(a, b);
}

console.log(calculate(10, add, 5));       // 15
console.log(calculate(10, subtract, 3));  // 7
console.log(calculate(4, multiply, 6));   // 24
console.log(calculate(10, divide, 2));    // 5
console.log(calculate(10, divide, 0));    // "Error: division by zero"

// 3. Rest parameters
function sumAll(...nums) {
  return nums.reduce((total, n) => total + n, 0);
}

console.log(sumAll(1, 2, 3, 4, 5));     // 15
console.log(sumAll(10, 20, 30));         // 60
console.log(sumAll(...[1, 2, 3, 4, 5])); // 15 — spread + rest working together!

// 4. Closure logger
function makeLogger(prefix) {
  return function(message) {
    console.log(`[${prefix}] ${message}`);
  };
}

const appLog  = makeLogger("APP");
const errLog  = makeLogger("ERROR");

appLog("Server started");    // "[APP] Server started"
errLog("Connection failed"); // "[ERROR] Connection failed"
appLog("Request received");  // "[APP] Request received"

// 5. Default parameters
function greet(name = "stranger", greeting = "Hello") {
  return `${greeting}, ${name}!`;
}

console.log(greet("Naman", "Hey")); // "Hey, Naman!"
console.log(greet("Priya"));        // "Hello, Priya!"
console.log(greet());               // "Hello, stranger!"
